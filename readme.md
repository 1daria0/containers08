# IWNO8: Непрерывная интеграция с помощью Github Actions

## Цель работы

В рамках данной работы студенты научатся настраивать непрерывную интеграцию с помощью Github Actions.

## Задание

Создать Web приложение, написать тесты для него и настроить непрерывную интеграцию с помощью Github Actions на базе контейнеров.

- Создан репозиторий на GitHub: `containers08`.
- Репозиторий склонирован на локальную машину.

### 3. Разработка Web-приложения

- **`config.php`** – содержит путь к SQLite-базе данных.
- **`Database`** – реализует все требуемые методы.
- **`Page`** – реализует метод `Render`.
- **`index.tpl`** – HTML-шаблон страницы.
- **`style.css`** – базовые стили.
- **`index.php`** – точка входа.

### 4. База данных

Файл `sql/schema.sql` создаёт таблицу `page` и заполняет её тремя тестовыми записями. При сборке Docker-образа этот файл используется для инициализации БД.

### 5. Написание тестов

- **`testframework.php`** – предоставляет простой фреймворк для тестирования.
- **`tests.php`** – содержит тесты для всех методов.

### 6. Dockerfile

Создан Docker-образ на основе `php:7.4-fpm`. В процессе сборки:
- Устанавливаются `sqlite3` и расширение `pdo_sqlite`.
- Создаётся том `/var/www/db`.
- Инициализируется база данных из `schema.sql`.
- Копируются файлы приложения из `site/` в `/var/www/html`.

### 7. GitHub Actions

Файл `.github/workflows/main.yml` настроен на запуск при push в ветку `main`. Workflow выполняет следующие шаги:
1. Клонирование репозитория.
2. Сборка Docker-образа `containers08`.
3. Создание контейнера с томом `database`.
4. Копирование тестов в контейнер.
5. Запуск контейнера.
6. Выполнение тестов командой `php /var/www/html/tests/tests.php`.
7. Остановка и удаление контейнера.

После каждого push в `main` тесты запускаются автоматически. Результат можно посмотреть во вкладке **Actions** репозитория.

### 8. Результат

Все 9 тестов проходят успешно. Это означает, что приложение работает корректно.


## Ответы на вопросы

**1. Что такое непрерывная интеграция?**  
Непрерывная интеграция (CI) – это практика, при которой разработчики часто объединяют свои изменения в общий репозиторий, после чего автоматически выполняются сборка и тестирование. Это помогает быстро находить ошибки и поддерживать код в рабочем состоянии.

**2. Для чего нужны юнит-тесты? Как часто их нужно запускать?**  
Юнит-тесты проверяют отдельные компоненты программы (функции, методы). Они нужны, чтобы убедиться в корректности кода после изменений. Запускать их следует при каждом изменении кода (локально перед коммитом) и обязательно в CI при каждом push или pull request.
